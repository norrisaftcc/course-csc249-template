Hash Tables

From convention check-in lines to password security — one data structure, two applications.

Motivation

One line is slow

Picture a convention check-in with 500 attendees and a single line. Everyone waits behind everyone else.

This is O(n) — the time to find your badge scales linearly with the number of people.

The obvious fix: split into multiple lines. But how you split them matters.

The question isn't "should we have multiple lines?" — it's "what rule decides which line you go to?"

That rule is a hash function.

Analogy

Check-in lines are hash tables

A standard convention system: two lines sorted by last name.

A–J →
Anderson Banks James Johnson
K–Z →
Kristol Young Zappa

Each line is a linked list. The collection of lines is an array. Together: a hash table.

A bad split makes the structure useless:

A–Y →
Anderson Banks James Johnson Kristol Young
Z →
Zappa

Unbalanced bins = one line does all the work. This is the hash table equivalent of a bad hash function.

Design

The fundamental tradeoff

How you divide your bins determines two things:

  • Wasted bins — empty lines with nobody in them (wasted memory)
  • Long chains — too many items in one bin (slow lookups)

Would 26 lines (one per letter) be 26× faster? Not if there's a Johnson Family Reunion — the J line would be longer than all the rest combined.

A good hash function distributes items evenly across bins. That's the whole game.

This tradeoff — memory vs. speed, bin count vs. chain length — is the core design problem of every hash table implementation.

Structure

Hash Table = Array of Linked Lists

[0] →
apple avocado
[1] →
banana
[2] →
cantaloupe cherry coconut
[3] →
∅
[4] →
elderberry

Each key goes through a hash function that returns an index. The item is stored in the linked list at that index.

Works with any key type — strings, numbers, objects — as long as you can hash it to an integer.

When two keys hash to the same index (like "apple" and "avocado"), that's a collision. The linked list at that bin handles it by chaining.

Performance

Average-case access time

O(1)

When collisions are rare, accessing an item by key takes roughly the same time as an array lookup — regardless of how many items are stored.

Compare to alternatives:

StructureLookup
Unsorted arrayO(n)
Sorted arrayO(log n)
Hash table (avg)O(1)
Hash table (worst)O(n)

Worst case: every key hashes to the same bin. This is why hash function quality matters.

Implementation

Simple hash functions

Name → line (convention style)

// 2 bins: A-J and K-Z
map(name) → first_letter < 'K' ? 0 : 1

2 bins = 2 linked lists. O(n/2) is still O(n), but for a finite crowd, twice as fast beats nothing.

Student ID → bin (modulo style)

// 10 bins using modulo
map(student_id) → student_id % 10

Works with any size ID. But if all IDs end in zero, every student lands in bin 0. The hash function must match your data distribution.

Implementation

Handling collisions

Chaining

Collisions are stored in the same bin, in a linked list. Walk the chain to find your item.

This is what we focus on in this class — it's the easiest to understand and implement.

[3] →
key_A key_B key_C

Open addressing (probing)

If a collision would occur, probe for a different empty bin.

  • Linear probing: try bin+1, bin+2, ...
  • Quadratic probing: try bin+1², bin+2², ...

Different implementation, same interface. You'll encounter these in job interviews.

Workforce

You already use hash tables daily

In Python, a dict is a hash table. In C++, std::unordered_map is a hash table.

# Python — this is a hash table
definition_of = {}
definition_of["doe"]  = "a deer, a female deer"
definition_of["far"]  = "a long long way to go"
// C++ — this is also a hash table
unordered_map<string, string> definition_of;
definition_of["doe"]  = "a deer, a female deer";
definition_of["far"]  = "a long long way to go";

The older name for this pattern is associative array — an array where the index is a string instead of a number. The hash function converts that string to an actual array index under the hood.

Application

Hashing in Cryptography

Same concept, different application. The hash function that assigns bins can also protect passwords.

Security

No system is 100% secure

Databases get breached. It happens to companies of every size. The question isn't whether your user database will ever be compromised — it's what the attacker gets when it is.

If the database stores passwords in cleartext (unencrypted), the attacker now has:

  • Usernames (typically email addresses)
  • The exact password each person used on your site

Since people reuse passwords, the attacker can now try those email + password combinations on banking sites, email providers, and everything else.

The 2015 Ashley Madison breach leaked customer billing information. Users who expected anonymity were blackmailed. Cleartext storage made it worse than it had to be.

Security

Password hashing: store the hash, not the password

Hash functions have a critical property: they are one-way.

  • Hashing a key always returns the same hash value
  • But the hash value is not enough information to recover the original key

This means you can answer "did they type the correct password?" without ever storing the password itself.

Register: hash the password, store the hash.
Login: hash what they typed, compare to the stored hash.
Match? Correct password. No match? Wrong password.
At no point is the actual password stored anywhere.

Security

The verification flow

Python

username = input("username: ")
login_pw = input("password: ")

login_hash = hash_password(login_pw)
stored_hash = db.get_hash(username)

if login_hash == stored_hash:
    correct_login = True

C++

string username, login_pw;
cin >> username >> login_pw;

auto login_hash =
    hash_password(login_pw);
auto stored_hash =
    db.get_hash(username);

if (login_hash == stored_hash) {
    correct_login = true;
}

The password never touches the database. Only the hash does.

Attack

The rainbow table problem

Hash functions are one-way, but an attacker can hash millions of common passwords and build a lookup table:

PasswordHash
123456e10adc3949...f883e
password5f4dcc3b5a...b8327
qwertyd8578edf84...5ca4
dragon8621ffdbc5...3db3

If your stored hash matches any entry in this table, the attacker knows your password. And if 500 users all chose "123456", cracking one cracks all 500.

This is the collision problem from hash tables, applied to security: identical inputs produce identical outputs. We need a way to make identical passwords produce different hashes.

Defense

Hashing & Salting

Add a pinch of randomness so identical passwords never produce identical hashes.

(but not smothered and covered)

Defense

How salting works

Registration:

  • User enters username + password
  • Generate a random salt string
  • Concatenate: password + salt
  • Hash the combined string
  • Store the hash and the salt

Login:

  • User enters their password
  • Load the stored salt for that user
  • Concatenate: password + salt
  • Hash the combined string
  • Compare to stored hash

The salt is not secret — it's stored right next to the hash. Its purpose isn't secrecy. Its purpose is uniqueness: every user gets a different salt, so identical passwords produce different hashes.

Defense

Salting in action

User "bob" registers with password "123456":

"123456" + "$.Qwe4k" → hash → a7f29c...

User "carol" registers with the same password "123456":

"123456" + "mK9#xPz" → hash → 3e81d4...

Same password. Different salts. Completely different hashes.

An attacker who cracks bob's hash learns nothing about carol's account.

Defense

Why salting defeats rainbow tables

Without salting, an attacker precomputes hashes for common passwords once, then matches against every user. One table cracks millions of accounts.

With salting, the attacker would need to run the hash function on every combination of:

  • Every possible password they might use
  • × Every possible salt

With an 8-character alphanumeric salt (a–z, A–Z, 0–9 = 62 characters), that's 62⁸ ≈ 218 trillion additional hash computations per password guess.

Modern production systems use hash functions like bcrypt and Argon2 that are intentionally slow to compute — adding maybe one second per login attempt. That's nothing for a real user. But for an attacker trying billions of combinations, it's a wall.

One concept, two applications
